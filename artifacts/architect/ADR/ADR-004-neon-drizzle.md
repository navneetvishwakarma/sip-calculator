# ADR-004: Drizzle ORM + Neon Serverless Postgres

Date: 2026-05-12
Status: Accepted

## Context

Phase 6 requires durable storage for two concerns: user sessions (managed by Better Auth) and saved scenarios. The schema is relational — one user has many scenarios; each scenario has typed inputs (`ScenarioParams` JSONB). The database must integrate cleanly with Next.js App Router route handlers running on Vercel's serverless runtime, where connection lifetime is per-request (not persistent). Two decisions are coupled: the database provider (managed Postgres vs. alternatives) and the ORM/query layer (how the app talks to the DB).

Prisma's Rust-based query engine adds binary size and cold-start overhead that compounds in serverless environments — each cold start loads the Prisma engine binary. Drizzle is implemented in TypeScript, has no native binary, and generates queries that run directly through the database driver. For a two-table schema owned by one engineer, Drizzle's lower abstraction level is an advantage, not a liability. Neon provides serverless Postgres with a `@neondatabase/serverless` driver designed for connection-per-request usage, plus a first-party Vercel integration that auto-provisions `DATABASE_URL`.

## Decision

Drizzle ORM (`drizzle-orm@^0.40`) with `drizzle-kit` for migrations, connected to Neon serverless Postgres via `@neondatabase/serverless`. The database schema is defined in TypeScript in `packages/db/src/schema.ts` (or `apps/web/src/db/schema.ts` if no shared DB package is needed before React Native v2). Migrations are generated by `drizzle-kit generate` and applied by `drizzle-kit migrate`. Better Auth's Drizzle adapter runs its own migrations for the auth tables separately. The `scenarios` table is the only app-owned table.

## Consequences

- Drizzle queries are SQL-shaped TypeScript: `db.select().from(scenarios).where(eq(scenarios.userId, userId))`. Engineers comfortable with SQL can read Drizzle queries directly — no ORM-specific query API to learn.
- No Rust binary in the bundle. Cold starts on Vercel serverless functions are faster than with Prisma.
- Drizzle's TypeScript inference provides full type safety on query results — `scenarios` rows are typed, not `any`.
- `drizzle-kit` generates SQL migration files that are committed to source control and inspectable. Migrations are explicit, not auto-applied on startup.
- Drizzle has fewer community tutorials than Prisma (as of 2025). A solo engineer debugging an unfamiliar Drizzle behaviour may spend more time on documentation. This is an acceptable trade for the cold-start advantage.
- Neon's Vercel integration handles `DATABASE_URL` injection; no manual secret management needed at deploy time.
- Neon's free tier (Hobby) is sufficient for v1 traffic. Upgrade to Neon Pro when connection limits are approached.
- `schema_version` field on the `scenarios` table is mandatory from day one. If `ScenarioParams` shape changes in a future phase, old rows with `schema_version: 1` can be migrated via a per-row function rather than a destructive ALTER TABLE. Omitting this field makes old saved scenarios silently unloadable.

## Alternatives Rejected

| Alternative | Reason Rejected |
|---|---|
| Prisma + Neon | Rust query engine binary; heavier cold starts on Vercel serverless; more abstractions than needed for a two-table schema |
| Drizzle + PlanetScale | PlanetScale is MySQL, not Postgres; no JSON operators for `inputs` column; JSONB is a Postgres strength |
| Drizzle + Supabase | Supabase free tier is generous but its extras (realtime, storage, edge functions) are dead weight for this app; Neon is leaner |
| Drizzle + Railway Postgres | More ops burden; no Vercel-native integration; cold starts on Railway's container model are different |
| Raw `node-postgres` (pg) | No TypeScript inference on query results; no migration runner; more boilerplate for every query |
| Knex | No TypeScript inference; weaker type safety than Drizzle; legacy feel in 2025 ecosystem |
